# **🔬 운빨 아발론 기술 진단 및 아키텍처 최적화 설계 제안서**

본 진단서는 제공된 소스 코드(index.html, game.js) 및 1차 점검 보고서의 데이터를 기반으로 작성되었습니다. 브라우저 기반 5대5 실시간 전장 환경에서 발생하는 프레임 드랍(Frame Drop)을 차단하고 스킬 수치 계산 방식을 체계화하기 위한 구체적인 가이드라인을 제시합니다.

## **📌 1\. 후반 성능 저하 및 프레임 드랍 핵심 원인 분석**

### **🚨 원인 A \- 이펙트 객체 배열의 무한 누적 및 비선형적 루프 연산**

* **현상** \- 게임이 중후반부로 진입하고 플레이어 및 AI 영웅들의 스킬 대기시간(Cooldown)이 감소하면서 화면에 발생하는 파티클(particles), 링(ringEffects), 광선(beamEffects) 등의 개수가 기하급수적으로 증가합니다.  
* **구조적 결함** \- game.js 내에 정의된 특수효과 배열들은 생성 시 제한 없이 push되지만, 수명이 다해 자연 소멸하는 속도보다 생성되는 속도가 빠르면 배열의 크기가 수백 개 이상으로 비대해집니다. 이를 매 프레임마다 무조건적으로 순회(forEach)하며 렌더링하기 때문에 CPU 연산 부하가 감당할 수 없을 만큼 상승합니다.

### **🚨 원인 B \- HTML5 Canvas의 shadowBlur 연산으로 인한 GPU/CPU 병목**

* **현상** \- 투사체나 장판 이펙트 표현 시 시각적 화려함을 더하기 위해 사용된 ctx.shadowBlur 및 ctx.shadowColor 연산이 매 프레임 수백 번 호출되고 있습니다.  
* **구조적 결함** \- HTML5 Canvas 2D 컨텍스트에서 그림자 블러(shadowBlur) 연산은 고비용의 소프트웨어 가우시안 블러 필터를 실행시킵니다. 그래픽 가속이 미비한 일반적인 웹 브라우저 환경에서는 이 연산 하나만으로도 프레임 레이트가 싱글 디지트(10fps 이하)로 수락하는 주범이 됩니다.

### **🚨 원인 C \- 화면 영역 외 렌더링 낭비 (Frustum Culling 부재)**

* **현상** \- 플레이어가 바라보고 있는 카메라 화면(Viewport) 영역 밖에서 일어나는 라인 미니언들의 전투, 포탑 사격, 중립 몬스터들의 특수 효과까지 모두 Canvas 드로잉 함수를 거치고 있습니다.  
* **구조적 결함** \- 화면 밖의 객체들은 위치 계산(물리 엔진 업데이트)만 수행하고 그리기(Draw) 연산은 생략해야 자원을 아낄 수 있으나, 현재는 모든 객체에 대해 무차별적인 렌더링이 호출되어 그래픽 파이프라인이 낭비되고 있습니다.

## **📌 2\. 스킬 수치 계산 오류 및 시스템 불안정성 진단**

### **⚠️ 위험 요소 A \- 쿨타임 감소율(CDR) 한계 결여로 인한 메인 스레드 정지**

* **현상** \- 로그라이크 강화 카드나 아이템 수치가 누적되어 쿨타임 감소 수치가 100%에 근접하거나 초과할 경우, 스킬 내부 쿨타임이 0초 이하로 연산되어 한 프레임 내에 무한 스팸 루프가 발생합니다.  
* **결과** \- 브라우저가 연산 처리를 끝내지 못하고 스크립트 실행 불가 상태(Freeze)에 빠져 게임이 강제 종료됩니다.

### **⚠️ 위험 요소 B \- 스킬 계수 연산 방식의 파편화 및 안전 장치 부재**

* **현상** \- 물리 공격력(ATK) 비례 데미지와 주문력(AP) 비례 데미지가 각 스킬 함수 내부에서 하드코딩된 방식으로 중구난방 연산되고 있습니다.  
* **결과** \- 레벨업이나 스탯 보정이 적용될 때 부동 소수점 오차나 계산 불능(NaN) 값이 발생하여 캐릭터가 즉사하거나 데미지가 들어가지 않는 무작위 버그의 원인이 됩니다.

## **📌 3\. AI 비서(클로드/안티그래비티)를 위한 구체적인 리팩토링 지시서 (의사코드 포함)**

안티그래비티 대화창에 아래의 수정안을 순차적으로 복사하여 리팩토링을 요청하십시오.

### **🛠️ \[지시서 1\] 이펙트 배열의 최대 제한 장치(Capping) 구축**

**요청 메시지**

"game.js에서 사용되는 시각 효과 배열들의 최대 개수를 제한하는 제어 장치를 추가해 주세요. 새로운 이펙트가 추가될 때 배열의 크기가 정해진 상한선을 넘으면 가장 오래된 이펙트(인덱스 0)를 즉시 제거(shift)하여 배열 크기가 일정 수준을 넘지 않도록 관리해 주시기 바랍니다."

**구현 기준 의사코드**

// 이펙트 종류별 최대 수치 설정  
const MAX\_EFFECT\_LIMITS \= {  
    particles \- 100,  
    ringEffects \- 20,  
    aoeEffects \- 15,  
    beamEffects \- 15,  
    textFloats \- 15  
};

// 안전한 이펙트 푸시 래퍼 함수 도입 예시  
function safePushEffect(arrayName, effectObject) {  
    const targetArray \= window\[arrayName\];  
    if (\!targetArray) return;  
      
    if (targetArray.length \>= MAX\_EFFECT\_LIMITS\[arrayName\]) {  
        targetArray.shift(); // 오래된 효과 제거로 메모리 확보  
    }  
    targetArray.push(effectObject);  
}

### **🛠️ \[지시서 3\] 뷰포트 기반 렌더링 생략 (Frustum Culling) 구현**

**요청 메시지**

"게임 내 카메라의 영역(카메라 중심 좌표 및 화면 가로/세로 폭)을 기준으로, 현재 카메라 화면 범위 밖에 위치한 미니언, 투사체, 타워, 이펙트 객체들은 물리 엔진 수치 업데이트만 진행하고 draw() 함수 호출은 건너뛰도록(Culling) 코드를 수정해 주세요."

**구현 기준 의사코드**

function isInsideCamera(obj, camera, padding \= 100\) {  
    return (  
        obj.x \>= camera.x \- padding &&  
        obj.x \<= camera.x \+ camera.width \+ padding &&  
        obj.y \>= camera.y \- padding &&  
        obj.y \<= camera.y \+ camera.height \+ padding  
    );  
}

// 렌더링 루프 내부 수정 예시  
entities.forEach(entity \=\> {  
    entity.update(); // 물리 연산은 항상 진행  
    if (isInsideCamera(entity, camera)) {  
        entity.draw(ctx); // 화면 안에 있을 때만 렌더링 실행  
    }  
});

## **💡 4\. 개발 성공률을 높이기 위한 핵심 꿀팁 (Tip)**

**AI 어시스턴트 협업 꿀팁**

안티그래비티나 클로드 같은 도구를 활용하여 코드를 개선할 때, \_\_전체 소스 코드를 한 번에 전부 올려서 고쳐달라고 지시하는 방식\_\_은 절대로 피하셔야 합니다. 대용량 코드를 한 번에 처리하려 하면 AI가 코드의 중요한 분기점이나 주요 로직을 빠뜨려 오히려 게임이 아예 실행되지 않는 버그가 생기기 쉽습니다.

수행률을 극대화하려면 위 가이드라인에 나열된 순서대로 \_\_모듈별 순차적 리팩토링\_\_을 진행하는 것이 좋습니다.

* 1단계 \- "이펙트 배열 개수 제한 장치를 game.js의 관련 배열들에 먼저 적용해 줘."  
* 2단계 \- "위 작업이 끝났으니, 다음으로 game.js 안에서 ctx.shadowBlur 코드를 찾아서 성능에 지장이 없도록 지우거나 대체해 줘."

위와 같이 하나씩 점진적으로 수정하며 직접 테스트를 거치는 방식이 가장 안전하고 빠르게 밸런스를 잡는 비결입니다.
